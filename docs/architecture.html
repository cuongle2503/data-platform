High-Level System Architecture

The Intelligent Data Platform (IDP) is an on-premise-first, single-node optimized data ecosystem based on the Medallion Data Architecture. It orchestrates economic data from the World Bank into an AI-ready intelligence layer.

Core Technology Stack

Data Persistence

  • MinIO Object Storage for raw, immutable Bronze data (Parquet). S3-compatible.
  • PostgreSQL 16 Relational DB for structured Gold dimensional models (Star schema).
  • pgvector Vector DB extension for storing document and metadata embeddings.

Processing & Intelligence

  • DuckDB OLAP Engine for in-process, high-speed data transformations without network overhead.
  • dbt-core Transformation Framework handling SQL modeling, testing, and dependency graphs.
  • Gemini API LLM/Embeddings for parsing intent, text embeddings (`text-embedding-004`), and RAG synthesis.

Application & Ops

  • Airflow 2.10.3 Orchestration managing DAGs across all 5 layers with retries and alerts.
  • FastAPI API Gateway serving REST endpoints with Pydantic validation.
  • Docker Compose Infrastructure for one-click single-node deployment.

Layer Specifics (Inputs & Outputs)

The system enforces strict boundaries between layers. Business logic is deferred to Silver/Gold, while Bronze remains an exact historical replica of the source.

1. Ingestion Layer (Bronze)

Focuses purely on extraction, pagination, and resilience. No business transformations.

Inputs • WB Indicators API (JSON)
• WB Documents API (JSON + Text)
• Rate: ~30 req/s
Process Python httpx extractors with exponential backoff. Converts JSON directly to Parquet using pyarrow to preserve schema.
Outputs Append-only Parquet files in MinIO:
s3://bronze/world_bank/indicators/year={YYYY}/data.parquet

2. Transformation Layer (Silver/Gold)

Executes data quality checks, standardization, and dimensional modeling.

Inputs Bronze Parquet files directly read from MinIO by DuckDB via S3 extension.
Process dbt compiles SQL. DuckDB executes in-memory. Casts types, handles NULLs, creates surrogate keys, builds facts/dimensions.
Outputs • Silver: Cleaned staging views.
• Gold: fct_economic_indicators, dim_country, dim_indicator.

3. Storage Layer (Serving)

Materializes data for high-performance concurrent querying and vector search.

Inputs • DuckDB Gold Models (via Python exporter).
• Text chunks from Silver.
Process Python scripts load data into PostgreSQL using INSERT ... ON CONFLICT (Idempotent). Generates Gemini embeddings for text.
Outputs • Indexed Postgres Tables.
• document_embeddings table with vector(768) columns (HNSW indices).

4. Intelligence Layer (RAG)

Interprets user intent, retrieves context, and synthesizes answers.

Inputs User natural language queries (e.g., "Why did Vietnam's GDP drop in 2021?")
Process 1. Intent Parsing (Query → SQL filters + Vector query).
2. Hybrid Search (pgvector similarity + structured SQL).
3. Prompt Assembly & Gemini Generation.
Outputs Structured JSON response containing the synthesized answer, exact data points used, and document citations.

The AAA Pipeline Pattern

All data flows adhere to the following strict principles to ensure reliability:

1. Idempotency First

Airflow tasks can fail and restart safely. MinIO files are overwritten (not duplicated). Postgres uses INSERT ... ON CONFLICT DO UPDATE. No duplicate rows, ever.

2. Out-of-Memory Processing

DuckDB streams Parquet files from MinIO to avoid RAM exhaustion on the 16GB server. Aggregations happen on disk/spill when necessary.

3. RAG Context Windowing

Documents are chunked into ~1500 character segments with overlap during extraction (Layer 1), ensuring the Gemini context window receives precise, targeted text rather than entire PDFs.
